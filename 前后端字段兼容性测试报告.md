## 一、问题严重等级说明

| 等级 | 含义 | 影响 |
|------|------|------|
| 🔴 P0-致命 | 字段名/路径完全不匹配，接口无法调用 | 功能完全不可用 |
| 🟠 P1-严重 | 字段类型不一致，运行时反序列化失败 | 功能报错 |
| 🟡 P2-中等 | 字段缺失或语义不一致 | 功能降级或数据丢失 |
| 🟢 P3-轻微 | 命名风格不一致但可兼容 | 维护成本 |

### 2.9 状态字段类型不一致汇总（🟠 P1-严重）

**核心问题**：后端不同实体的状态字段类型不统一，前端难以统一处理。

| 实体 | 后端字段类型 | 枚举值 | 前端期望类型 |
|------|-------------|--------|--------------|
| **用户状态** | `String` | `"ACTIVE"` / `"DISABLED"` | `1 \| 0` (数字) |
| 角色状态 | `Integer` | `1` / `0` | `1 \| 0` (数字) ✅ |
| 组织状态 | `Integer` | `1` / `0` | `1 \| 0` (数字) ✅ |
| 菜单可见性 | `Integer` | `1` / `0` | `1 \| 0` (数字) ✅ |
| 岗位状态 | `Integer` | `1` / `0` | `1 \| 0` (数字) ✅ |
| 字典状态 | `Integer` | `1` / `0` | `1 \| 0` (数字) ✅ |
| 配置状态 | （无） | - | - |
| 通知模板状态 | `Integer` | `1` / `0` | `1 \| 0` (数字) ✅ |
| 登录日志状态 | `Integer` | `1`成功 `0`失败 | `1 \| 0` (数字) ✅ |

**结论**：仅用户状态是字符串枚举，其余都是数字。前端 `EnableStatus = 1 | 0` 与用户状态不兼容。

### 2.12 边界情况测试（🟡 P2-中等）

**问题 19：空值处理**

| 场景 | 前端处理 | 后端处理 | 风险 |
|------|----------|----------|------|
| 用户名为空 | 表单校验拦截 | `@NotBlank` 校验 | ✅ 双重保障 |

| 邮箱格式错误 | **无前端校验** | `@Email` 校验 | 🟡 后端报错，前端未提示 |

**问题 23：字段级错误处理不完整**

## 四、解决方案

### 4.3 推荐方案选择

| 对比项 | 方案一（改后端） | 方案二（改前端） |
|--------|------------------|------------------|
| 修改文件数 | ~15 个后端文件 | ~20 个前端文件 |
| 影响范围 | 后端 API 契约变更 | 前端 UI 与类型变更 |
| 数据库影响 | 无（Entity 不变） | 无 |
| RESTful 规范 | ✅ 符合 | ❌ 不符合 |
| 推荐度 | ⭐⭐⭐⭐⭐ | ⭐⭐ |

**最终建议**：采用**方案一（改后端）**，理由：
1. 前端字段命名更规范（`username` 比 `account` 通用）
2. 状态字段统一为数字（`1/0`）比字符串枚举更简洁
3. 后端通过 DTO/VO 层隔离 Entity，修改成本低
4. 避免修改大量前端 Vue 组件和 TypeScript 类型
